# 自我反思与改进

## 一、这次做错了什么

### 1. 弹簧物理（#24）反复失败 — 最大的教训

用户要求"拖拽父节点时，子节点跟随运动"，我修了 **5 次** 才勉强做对：

| 次数 | 方案 | 失败原因 |
|------|------|----------|
| 第1次 | requestAnimationFrame + left/top 定位 | CSS 浮动动画用 `transform: translate()`，与 left/top 冲突，子节点视觉上不动 |
| 第2次 | 把函数从闭包移到模块级 | 没解决根本问题（CSS 冲突），只是重构了代码结构 |
| 第3次 | transform + 速度弹簧物理 | 速度衰减参数不对，快速拖拽时子节点位置错乱 |
| 第4次 | 简单 lerp + 双频飘动 | lerp 系数太小（0.02~0.05），运动太迟钝 |
| 第5次 | 加大 lerp + 三层频率叠加 | 用户还在反馈不够飘逸，参数仍在调优中 |

**根本原因：** 我没有在动手前分析 CSS 动画系统。项目中有三个动画（float1/float2/float3）都用 `transform: translate()`，而我试图用 `left/top` 移动节点——这两套系统互不干扰，但 CSS 动画会"盖住"left/top 的效果。我应该第一件事就读 style.css，理解动画冲突再写代码。

### 2. 功能遗漏 — 没有逐条核对需求

用户给了 25 条需求文档，我声称"都实现了"，但实际上：
- **#24 弹簧物理** — 完全没实现，用户追问才发现
- **#25 折叠/展开** — 数据字段存在但没有 UI 入口（徽章不可点击）

**根本原因：** 我没有拿着需求文档逐条检查代码。我凭"看起来都有"的印象做了判断，而不是 grep 每个功能点的实现。

### 3. Prompt 优化方向偏差

第一次优化 Prompt 时，我写了一堆抽象、隐喻式的联想方向（"如果 X 是一种情绪"），用户直接否了——他是设计师，要的是具体、有网感、能落地的词。

**根本原因：** 我没有先理解用户的使用场景。"创意发散"对设计师来说是找灵感素材，不是写诗。

## 二、为什么会犯这些错

### 思维模式问题

1. **急于动手，不先分析** — 看到"弹簧物理"就开始写 requestAnimationFrame，没有先读 CSS 理解动画系统
2. **自信过头，不验证** — 写完代码就说"实现了"，没有逐条对照需求检查
3. **假设用户意图，不确认** — Prompt 优化时假设用户要"文艺感"，实际要"网感+具体"
4. **修 bug 时打补丁，不重构** — 弹簧物理每次都在上一个错误方案上修补，而不是推倒重来分析根因

### 流程问题

5. **没有先读完所有相关文件** — 改 graph.js 前应该先读 style.css，理解 CSS 动画如何影响节点定位
6. **没有用需求文档当 checklist** — 应该建一个表格，每条需求对应"已实现/未实现/有 bug"
7. **没有在第一次失败后停下来思考** — 弹簧物理第 2 次失败后就应该分析 CSS 冲突，而不是继续在同一思路上修

## 三、下次怎么改进

### 动手前

1. **先读完所有相关文件** — 改 A 文件前，先读 A 依赖的 B、C 文件，理解全貌
2. **需求文档当 checklist** — 拿到需求后建表格，逐条标记状态，完成一条勾一条
3. **理解现有系统再改** — 特别是 CSS/动画/事件系统，先画出"谁影响谁"的关系图

### 写代码时

4. **一个功能从最简方案开始** — 弹簧物理应该先做"子节点直接跟随父节点"（最简），确认能动，再加弹性、飘动
5. **每写完一个功能就测试** — 不要写完 5 个功能才测试，边写边验证
6. **CSS 动画和 JS 定位要统一方案** — 要么都用 transform，要么都不用，不要混用 left/top 和 transform

### 修 bug 时

7. **第一次失败后分析根因** — 不要在错误方案上打补丁
8. **加 console.log 验证假设** — 不要猜，要看实际运行时变量的值
9. **如果 2 次修不好，换思路** — 说明根因没找到，不是参数问题

### 和用户沟通时

10. **不确定的功能不要说"已实现"** — 说"我检查一下"比"都做好了"好
11. **先问使用场景** — 特别是 Prompt 类需求，先了解用户是谁、要什么效果
12. **用户说"不行"时，不要继续在同一方向调参数** — 应该问清楚具体哪里不满意

---

## 四、核心原则

> **先理解，再动手。先验证，再说完成。**
